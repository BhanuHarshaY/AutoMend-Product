- **Dev-machine Redis conflict** — Ray occupies port 6379 on the dev machine, so the app-intended Redis (also 6379) can't run alongside it. Tests work around this by using port 6380 for the test Redis. Running the full Docker Compose stack on such a machine requires stopping Ray first. Documented in CLAUDE.md §10.
- **Temporal test environment on Windows** — the workflow-failure-propagation test path hangs when a workflow raises an uncaught exception. Worked around by testing checksum validation as a unit test rather than an end-to-end workflow test (DECISION-010).
- **pgvector IVFFlat indexes** — deferred to a post-seed migration (DECISION-003). The initial migration creates tables and the `vector` extension but not the indexes, since IVFFlat requires existing data to build its list structure. Add them via a follow-up migration after production seed data lands.

### Phase 11 bug fixes applied during the live manual-test walkthrough
> The first actual run of `MANUAL_TESTING.md` surfaced 15 distinct issues —
> most of them first-time kind + helm + Next.js gotchas that no amount of
> lint + unit testing would have caught. Every one is now fixed in-place
> with a permanent remedy committed to the chart or the images, plus a
> matching entry in `MANUAL_TESTING.md` §15. DECISION-025 captures the
> architectural takeaways.
- **Multi-arch image loading into kind** — Docker Desktop's containerd image store keeps manifest lists that trip `ctr import --all-platforms`. Fix: `docker save --platform linux/amd64` + `kind load image-archive` rather than `kind load docker-image`.
- **`runAsNonRoot` without explicit `runAsUser`** on backend pods — the `python:3.11-slim` base image's default USER is root; kubelet rejects. Added `runAsUser: 1000` / `runAsGroup: 1000` / `fsGroup: 1000` to every backend component's `podSecurityContext` in `values.yaml`.
- **Non-numeric `USER nextjs` in Dockerfile.frontend** — kubelet can't verify non-root without a UID. Added `runAsUser: 1001` to the frontend's podSecurityContext matching the UID the Dockerfile creates.
- **Kubernetes service env var injection clobbering `AUTOMEND_API_PORT`** — k8s injects `AUTOMEND_API_PORT=tcp://10.96.x.y:8000` into every pod in the namespace when a Service named `automend-api` exists; pydantic-settings then fails to parse `api_port: int`. Fix: `enableServiceLinks: false` on every pod spec (all 6 app Deployments + the migrations Job).
- **`readOnlyRootFilesystem: true` blocked Python pycache writes** — relaxed to `false` for backend containers in `values.yaml`; tighten via tmpfs mounts for production in Phase 12.
- **Postgres initdb scripts skipped on an existing PVC** — when postgres-dev was first installed without the `initdb` ConfigMap, the subsequent chart upgrade added the ConfigMap but postgres's entrypoint refused to re-run init scripts. Documented the "delete the PVC to re-run initdb" recovery in MANUAL_TESTING.md §15.
- **Temporal `auto-setup` image doesn't support SQLite** — originally configured with `DB=sqlite` which doesn't exist in that image. Rewritten `temporal-dev.yaml` to point auto-setup at the in-cluster postgres-dev with dedicated `temporal` + `temporal_visibility` databases created by postgres's initdb ConfigMap.
- **Alembic `ModuleNotFoundError: No module named 'app'`** — `Dockerfile.api` does `pip install .` before copying sources so the `app` package isn't registered as a module; uvicorn works because it adds CWD to sys.path, but alembic (a console_script) doesn't. Fix: `prepend_sys_path = .` in `backend/alembic.ini`, plus `PYTHONPATH=/app` in the migrations-job.yaml Job command as belt-and-braces for existing images.
